Idun Agent Platform uses the [Model Context Protocol (MCP)](https://modelcontextprotocol.io) to extend agent capabilities with external tools. You define MCP servers at the platform level, then attach specific servers to individual agents. Each agent can only access the tools from its assigned servers, so you control exactly what each agent can do.

<Tip>
  The [langgraph-tool-local template](/templates) shows how to mix local tools with MCP tools from the platform in a single agent.
</Tip>

## How tool governance works

1. **Register MCP servers** in the Manager UI or YAML config with transport settings and connection details
2. **Attach servers to agents** through per-agent allowlists, so each agent only sees the tools you assign
3. **The engine discovers tools** from each configured MCP server at startup
4. **Agents invoke tools** during conversations as needed, with results passed back for response generation

## Per-agent tool allowlists

Every agent has its own list of MCP servers. An MCP server registered in the platform is not available to an agent until you attach it. This means you can:

- Give a customer support agent access to a CRM tool but not a database query tool
- Let a research agent fetch web content but block file system access
- Share the same MCP server across multiple agents or restrict it to one

## Transport types

MCP servers connect to the engine through one of four transport protocols:

| Transport | Use case | Required fields |
|---|---|---|
| `stdio` | Local processes, Docker containers | `command`, `args` |
| `sse` | Remote servers with Server-Sent Events | `url` |
| `streamable_http` | Remote servers with HTTP streaming (default) | `url` |
| `websocket` | Persistent bidirectional connections | `url` |

## Configuration example

Define MCP servers in your `config.yaml` or through the Manager API:

<Tabs>
  <Tab title="Manager UI">
    <Steps>
      <Step title="Browse the transport catalog">
        Navigate to **MCP Servers** in the sidebar. The catalog shows transport type cards: Streamable HTTP, SSE, WebSocket, and STDIO.
        ![MCP server catalog](/images/ui/mcp-catalog.png)
      </Step>
      <Step title="Add an MCP server">
        Click **+** on your transport type and configure the command/URL and connection details. The page includes a Quick Start Guide and a "Discover tools" button on existing server configs.
        ![Create STDIO MCP server](/images/ui/mcp-create-stdio.png)
      </Step>
      <Step title="Attach to an agent">
        Attach the server to agents from the agent detail page.
      </Step>
    </Steps>
  </Tab>
  <Tab title="YAML config">
    ```yaml config.yaml
    mcp_servers:
      - name: fetch
        transport: stdio
        command: docker
        args: ["run", "-i", "--rm", "mcp/fetch"]

      - name: filesystem
        transport: stdio
        command: npx
        args: ["-y", "@modelcontextprotocol/server-filesystem", "/data"]
        env:
          ALLOWED_DIRECTORY: /data

      - name: custom-api
        transport: streamable_http
        url: https://mcp.example.com/api
        headers:
          Authorization: "Bearer ${MCP_API_TOKEN}"
    ```
  </Tab>
</Tabs>

## Framework integration

The engine provides helper functions to load MCP tools into your agent code:

<CodeGroup>
```python ADK
from idun_agent_engine.mcp import get_adk_tools

idun_tools = get_adk_tools()
```

```python LangGraph
from idun_agent_engine.mcp import get_langchain_tools

mcp_tools = await get_langchain_tools()
```
</CodeGroup>

These functions discover all MCP servers attached to your agent and make their tools available. You do not need to configure individual tools.

<Warning>
  `get_langchain_tools()` is an async function. Use `await` when calling it.
</Warning>

## Expose an agent as an MCP server

Any engine-hosted agent can also be consumed as an MCP tool by other agents and MCP clients. The engine serves a streamable HTTP MCP endpoint at `/mcp` that exposes the agent as a single tool named after the agent.

The tool's input schema is derived from the agent's declared capabilities:

- **Chat input** agents take `message` (string) and optional `session_id`.
- **Structured input** agents take an `input_data` object matching the agent's full JSON schema.

The tool's output respects the agent's output mode. For structured output, the engine returns the final state filtered to the keys declared in the output schema, falling back to the full state and then to streamed text. That covers both LangGraph agents that write results to state and ADK agents that return JSON in a text message.

### Enable

Exposure is enabled by default. Control it through the `server` section of your engine config:

```yaml
server:
  api:
    port: 8000
  as_mcp: true
  mcp_description: "Query the support knowledge base"
```

| Field | Type | Default | Purpose |
|---|---|---|---|
| `as_mcp` | boolean | `true` | Serve the `/mcp` endpoint |
| `mcp_description` | string, optional | null | Tool description shown to MCP clients. Falls back to `"Invoke the <agent-name> agent"` |

In the Manager UI, toggle **Expose as MCP Server** on the agent creation wizard or the agent detail page. When enabled, the platform registers a matching MCP server entry pointing at the agent's `/mcp` URL so you can attach it to other agents with one click.

### Consume the agent from another agent

Register the agent's endpoint as a `streamable_http` MCP server:

```yaml
mcp_servers:
  - name: support-agent
    transport: streamable_http
    url: https://support.example.com/mcp
```

When the manager runs in a container and the target agent runs on the host, use `host.docker.internal` (Mac and Windows) or the host's routable address instead of `localhost`.

Authentication follows the agent's [SSO config](/auth/overview). When SSO is enabled on the agent, MCP clients must present a valid bearer token through the `Authorization` header.
